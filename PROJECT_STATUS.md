# Midasbuy Redeem Service

## О проекте
Сервис для автоматической фоновой активации (redeem) UC кодов на платформе Midasbuy для игры PUBG Mobile. Архитектура построена на базе очередей задач (BullMQ) и реального браузера (Playwright) для обхода защиты Cloudflare и антифрод-систем.

---

## 🛠 Что уже сделано (Текущий статус)

### 1. Архитектура и База Данных
- Настроена база данных PostgreSQL через Kysely (query builder).
- Созданы таблицы для аккаунтов Midasbuy (`accounts`) и логов активации (`redeem_logs`).
- Настроен Redis для очередей задач и кеширования.
- Проект полностью переведен на современные **ESM модули** (`"type": "module"`) с использованием сборщика `tsup`, что решило проблемы совместимости с современными npm-пакетами.

### 2. Управление аккаунтами (Логин и Сессии)
- Реализован механизм автоматического входа в аккаунты Midasbuy через реальный браузер с помощью **Playwright**.
- Настроен обход базовой защиты Cloudflare при логине с использованием плагинов `puppeteer-extra-plugin-stealth`.
- Реализовано безопасное сохранение сессионных cookies в базу данных с их предварительным шифрованием (AES-256-GCM).
- Создан фоновый воркер (`session-refresh.worker.ts`), который следит за актуальностью сессий и автоматически их обновляет, если они протухли.

### 3. API и Очереди (Redeem Flow)
- Создан Fastify REST API сервер с валидацией входных данных через Zod.
- Настроен эндпоинт `POST /api/v1/redeem` для приема пачек UC кодов для конкретного Player ID.
- Реализована надежная очередь задач через **BullMQ** (`redeem.worker.ts`). Сервер моментально отвечает `202 Accepted` и ставит задачи в очередь, чтобы не блокировать клиент.
- Настроены механизмы повторных попыток (retries) с экспоненциальной задержкой (backoff) для временных ошибок.
- Настроен **Circuit Breaker** (предохранитель) с помощью библиотеки `opossum`, который останавливает запросы к Midasbuy, если их API начинает массово возвращать ошибки или блокировать нас.

### 4. Инфраструктура
- Настроен `docker-compose.yml` со всеми необходимыми сервисами (App, PostgreSQL, Redis).
- Написан оптимизированный `Dockerfile` с многоэтапной сборкой (Multi-stage build) для уменьшения размера итогового образа.

## ⚠️ Текущая проблема (На чем остановились)

В данный момент мы успешно авторизуемся и получаем валидные cookies. Однако, когда фоновый воркер пытается отправить сам код активации (Redeem) через HTTP-запрос (с использованием библиотеки `got-scraping`), **Cloudflare блокирует этот POST-запрос** и возвращает HTML-страницу с капчей.

Лог ошибки:
`{"level":"ERROR","msg":"Blocked by Cloudflare/Captcha","msg":"Failed to parse Midasbuy response"}`

Из-за этого срабатывает Circuit Breaker, и задачи переходят в статус `failed`.

## 🚀 Что нужно сделать дальше (План действий)

### 1. Перевод процесса активации (Redeem) на Playwright
Так как обычные HTTP-запросы (даже с подделкой TLS-отпечатков через `got-scraping`) блокируются Cloudflare на эндпоинте `QueryRedeemCodeInfo`, необходимо отказаться от этого подхода.

**Задача:** Переписать метод `executeRedeemRequest` в классе `MidasHttpClient` (или создать новый сервис).
- **Вместо HTTP-запроса:** Открывать скрытую страницу в Playwright (headless).
- **Действия:** 
  1. Подставлять актуальные cookies из БД в контекст браузера.
  2. Переходить на страницу активации Midasbuy.
  3. Делать POST-запрос через API самого браузера (`page.request.post()`) ИЛИ физически заполнять поля Player ID и Redeem Code на странице и кликать кнопку "ОК".
- Это позволит на 100% обходить защиту Cloudflare, так как запрос будет исходить из полноценного браузера, который уже прошел все проверки.

### 2. Обработка прокси
- Убедиться, что каждый аккаунт/запрос использует свой прокси-сервер (в Playwright это настраивается при создании контекста браузера `browser.newContext({ proxy: ... })`), чтобы избежать бана по IP за подозрительную активность.

### 3. Улучшение обработки ошибок Midasbuy
- После перевода на Playwright, нужно корректно парсить ответы сайта (например, "Код уже использован", "Неверный Player ID", "Лимит активаций") и записывать правильный статус в таблицу `redeem_logs`.

### 4. Мониторинг и метрики (Опционально)
- Прикрутить Prometheus/Grafana (библиотека `prom-client` уже есть в зависимостях) для отслеживания длины очереди BullMQ, процента успешных активаций и состояния Circuit Breaker.
